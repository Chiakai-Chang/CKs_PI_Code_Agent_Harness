# RATIONALE: trycua/cua 整合決策理由書

## 1. 決策背景
*   **來源專案**：https://github.com/trycua/cua
*   **核心功能**：Computer-Use Agent (提供 AI 操作電腦的能力，包含虛擬機管理、Cua Driver 背景操作)。
*   **評估日期**：2026-05-13

## 2. 戰略分析 (Technical TOWS)
我們在評估過程中發現了顯著的「戰略矛盾」：
*   **S (優勢)**：具備 Pi 缺失的 UI 自動化能力。
*   **W (劣勢)**：依賴 Docker/QEMU/Swift，體積巨大，環境配置極其複雜。
*   **T (威脅)**：上游專案處於早期 Beta，架構變動頻繁。

根據 **v3.7 導則**，此情境屬於 **ST 策略區間**：我們渴望其能力，但必須隔離其風險與重量。

## 3. 最終決策：🟢 Path 2 (輕量橋接 - 作為選配)
我們拒絕將其核心代碼進行「蒸餾」或「移植」至 Core 下，理由如下：
1.  **維護性優先**：原專案更新極快，改寫代碼將導致無法享受上游修正。
2.  **嚴禁臃腫**：強行整合會破壞 Harness 的「秒級安裝、零依賴」靈魂。
3.  **靈魂保真**：Cua 的能力依賴於硬體虛擬化，無法被脫水成純 Markdown。

## 4. 實施方案：cua-commander
*   **形式**：僅在 `pi-skills/optional` 中提供一個「遠端指揮官」技能。
*   **運作邏輯**：Pi 不包含 cua。用戶需自行安裝啟動 cua 服務，Pi 透過 MCP 協議或 `cua-cli` 進行「指令式連線」。
*   **維護計畫**：保持與 cua 官方 MCP 接口的相容性，不重寫任何內部邏輯。
