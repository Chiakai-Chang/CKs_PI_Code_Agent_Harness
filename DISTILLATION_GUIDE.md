# 🧪 功能蒸餾指南 (Distillation Guide) v2.0

這份文件記錄了本專案如何將外部強大的 AI 代理專案（如 `ECC`, `Understand-Anything`）「蒸餾」並「移植」到 `Pi` 平台的技術架構、核心原理與避坑指南。

---

## 核心挑戰與設計原則 (MECE 原則)
移植外部專案不只是檔案拷貝，必須解決以下四大維度問題：
1.  **環境解耦**：解決路徑失效、Windows/Unix 差異與編碼衝突。
2.  **人格保真**：確保複雜的 TS/JS 代理邏輯在轉化為 `.md` 後不喪失推理品質。
3.  **依賴一致性**：管理多個插件間的依賴衝突與版本對齊。
4.  **安全邊界**：防止提權操作導致的權限污染與 Git 信任威脅。

---

## 蒸餾標準流程 (The Industrial Distillation SOP)

### 第一步：架構解構與能力對照 (Structural Discovery)
不要只看代碼，要看它的「權力結構」。
*   **Skills (靈魂)**：分析操作流程（例如 `/understand` 從 Phase 0 到 7 的邏輯閘）。
*   **Agents (大腦)**：找出執行這些流程的「人格」。
    *   *關鍵工作*：將隱藏在外部專案代碼中的 Prompt 範本**實體化**為 Pi 可識別的 `.md` 檔案。
*   **Tools (手腳)**：對照外部工具與 Pi 原生工具（如 `Read` 對應 `read_file`）。
*   **依賴掃描**：確認是否需要 `pnpm` 或特定版本的環境，優先採用「零依賴」改寫。

### 第二步：橋接與路徑注入 (Bridge & Inject Pattern)
這是確保功能「全域可用」的核心發明。
*   **翻譯層 (Bridge Extension)**：建立 TypeScript 擴充功能，監聽 Pi 事件（如 `tool_call`）並調用外部腳本。
*   **動態路徑定位 (PI_HARNESS_ROOT)**：
    *   **原理**：在安裝階段，自動獲取本倉庫的「絕對路徑」。
    *   **實作**：將路徑寫入擴充功能的 `package.json` 或 `settings.json`。
    *   **好處**：AI 在系統任何角落執行，都能精準跳回 Harness 倉庫尋找子模組。

### 第三步：環境自癒與權限管理 (Self-Healing & Sanitization)
*   **智慧提權**：偵測權限不足時，主動問詢並透過 `RunAs` (Win) 或 `sudo` (Unix) 重啟。
*   **Git 信任自動化**：自動處理 `dubious ownership` 報錯，執行 `safe.directory` 修正。
*   **編碼強制化**：全流程強制 UTF-8，杜絕中文亂碼。
*   **版本檢查 (Version Parity)**：橋接器應具備基本探針，確認子模組版本與 Harness 預期一致。

## 人格重構技術：三階 Prompt 蒸餾法
為了防止 distilled agent 變弱，必須遵循：
1.  **邏輯骨幹提取**：不抄寫長篇大論，只提取其決策樹與驗證標準。
2.  **工具適配**：將原本的 Bash 指令轉化為 Pi 的 `@generalist` 或專屬 Tool 標籤。
3.  **邊界定義**：在 `.md` 開頭明確定義「禁止行為」與「強制流程」。

---

## 蒸餾後檢核表 (Post-Distillation Checklist)
- [ ] **路徑測試**：在非專案資料夾啟動 `pi`，測試指令是否仍能正確觸發。
- [ ] **權限測試**：確認提權安裝後，普通使用者權限是否仍能讀取 Skills。
- [ ] **衝突測試**：多個插件同時載入時，是否有重複的指令命名？
- [ ] **效能測試**：長文本 (192k) 模式下，大腦地圖生成是否會導致 OOM？

---

## 實戰心得：保持輕量、保持靈魂
*   **配置與代碼分離**：使用 `.example` 設定檔，透過腳本自動生成 local 設定，這也是 Harness 能在 3 分鐘內完成部署的秘訣。
*   **工具優先**：能用 Python 內建庫（如 `urllib`）解決的，就不要依賴外部 API。
