# 🧠 CK's Pi Code Agent Harness：核心設計理念與加強原理

這份文件詳細說明了本專案如何在原生的 `Pi Coding Agent` 之上，透過五大核心系統將其改造成「資深工程師等級」的開發環境。

---

## 1. 自動糾察隊系統 (ECC Hooks Bridge)
**「在 AI 闖禍之前，先由專家攔截。」**

*   **它的角色**：它是 Pi 助手的「良知」與「規範執行者」。
*   **運作原理**：利用 Pi 的擴充介面，我們在 AI 「打算執行指令 (Bash)」或「按下存檔 (Edit)」的**瞬間**插入了檢查程序。它會呼叫 `Everything Claude Code (ECC)` 的高品質檢查腳本進行「秒級審查」。
*   **加強了什麼**：
    *   **品質門檻 (Quality Gate)**：如果 AI 寫的代碼少了一個括號或是不符合規範，糾察隊會直接擋下並叫 AI 重寫。
    *   **安全防護**：防止 AI 意外刪除 `.env` 檔案、把金鑰推送到 Git 或是在 `main` 分支進行危險操作。
*   **潛在問題與處理**：
    *   **問題**：這類腳本通常依賴「絕對路徑」，換電腦或換資料夾就失效。
    *   **處理**：我們實作了「絕對路徑自動注入」。在安裝時，系統會自動定位本倉庫位置並寫入 Pi 的全域設定，確保糾察隊不管在哪裡都能找到「家」。

## 2. 專家直覺系統 (Superpowers - Native)
**「不再只是直覺反應，而是先思考、再選工具。」**

*   **它的角色**：它是 Pi 助手的「職業訓練」與「作業標準書 (SOP)」。
*   **運作原理**：本專案直接透過 **Native Mapping** 整合了 `obra/superpowers` 框架。這是一套「元指令 (Meta-instructions)」，強制 Pi 在每一輪對話開始前，先反思：*「這題是 UI 嗎？這題要寫測試嗎？」* 然後主動去技能庫 (Skills) 找最適合的工具。
*   **加強了什麼**：讓 AI 從「會寫代碼的機器」進化成「懂開發流程的專家」。它會主動發起 `/tdd` (測試驅動開發) 或 `/plan` (詳細規劃)，確保開發過程嚴謹且可追蹤。
*   **整合優勢**：透過原生映射，我們能 100% 同步上游最新的工程紀律與人格優化，不遺漏任何參考資料（References）。
*   **潛在問題與處理**：
    *   **問題**：AI 有時會變得太過「話多」或過度思考。
    *   **處理**：我們在 Rules 中定義了優先級，並引導 Pi 優先使用輕量化的本地指令來平衡效能。

## 3. 代碼圖譜系統 (Understand-Anything)
**「為 AI 裝上 GPS，秒懂數萬行代碼的結構。」**

*   **它的角色**：它是 Pi 助手的「導航儀」與「架構分析師」。
*   **運作原理**：基於 `Understand-Anything` 專案，透過 `/understand` 指令，Pi 會掃描整個專案並生成一個 `knowledge-graph.json`。這不是單純的文字搜尋，而是建立代碼之間的「邏輯連結圖」。
*   **加強了什麼**：
    *   **架構解讀 (Onboarding)**：利用 `/understand-onboard`，AI 能在幾秒內告訴你哪些檔案是核心邏輯、哪些是工具類、層級如何劃分。
    *   **邏輯追蹤**：當你問「改了這個 Function 會影響誰？」時，AI 能透過圖譜給出精準的答案，而非瞎猜。
*   **潛在問題與處理**：
    *   **問題**：大型專案掃描會消耗大量運算與時間。
    *   **處理**：我們整合了「增量更新 (Incremental Update)」與「忽略清單 (.understandignore)」，確保 AI 只專注於核心邏輯，避免掃描無用的第三方套件（node_modules）。

## 🌟 專家調度指南 (Strategic Agent Activation)
雖然本 Harness 整合了超過 50 位專業代理人 (ECC Agents)，但**「全量開啟」並非最佳實踐**。為了達到 Token 消耗與開發品質的完美平衡，建議遵循以下調度策略：

### 1. 核心五人組 (The Core 5) —— 預設活躍
這五位專家覆蓋了 90% 的日常開發，建議始終保持其在 `config.json` 中的路徑映射：
*   **`planner`**：負責複雜任務的拆解。
*   **`architect`**：負責重大架構決策。
*   **`coder`**：負責高品質的代碼實現。
*   **`code-reviewer`**：負責提交前的代碼審查。
*   **`tdd-guide`**：負責測試驅動開發流程。

### 2. 特種部隊 (Specialists) —— 按需激活
針對特定技術棧或特定階段，您可以手動或由主代理引導啟動：
*   **UI/UX 任務**：啟動 `a11y-architect` (無障礙) 或 `design-system-engineer`。
*   **安全審計**：在釋放版本前，手動調用 `security-reviewer`。
*   **環境修復**：遇到編譯報錯時，調用 `build-error-resolver`。

### 3. 如何優化效能？
*   **減少過度查詢**：如果只是簡單的 typo 修正，可以暫時切換至 `caveman` 模式，避免觸發重型品質門檻。
*   **善用 Auto-Compact**：確保您的 `pi` 設定了適當的壓縮閾值，配合 `llm-wiki` 進行知識沉澱，能讓長對話依然保持精準。
