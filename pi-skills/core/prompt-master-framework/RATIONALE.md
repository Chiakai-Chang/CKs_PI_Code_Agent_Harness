# RATIONALE: nidhinjs/prompt-master 整合決策理由書

## 1. 決策背景
*   **來源專案**：https://github.com/nidhinjs/prompt-master
*   **核心功能**：AI 提示詞優化系統（含 12 種專業模板、35 種提示模式、Token 效率審計）。
*   **評估日期**：2026-05-13

## 2. 戰略分析 (Technical SWOT/TOWS)
*   **S (優勢)**：具備強大的「元提示」能力，能顯著提升跨代理協作（如 Pi 呼叫其他 LLM 或工具）的指令精準度。
*   **O (機會)**：將其作為 Pi 的「提示詞翻譯官」，優化 Token 消耗並減少開發時的溝通摩擦。
*   **T (威脅)**：提示工程技巧隨模型演進快速變動（如 R1/O1 系列對傳統提示的需求不同）。

根據 **v3.7 導則**，這屬於 **SO 策略區間**：功能具備高度互補價值，且結構標準，應透過原生映射維持長期演進。

## 3. 最終決策：🟢 Path 1 (原生映射 - 透過 Submodule)
理由如下：
1.  **維護性優先**：透過子模組直接連結官方倉庫，確保 Pi 隨時擁有最新的提示模板與去重演算法，無需手動維護副本。
2.  **高保真體驗**：保留原始專案的 `references/` 與模板結構，確保 AI 在執行 `/prompt-master` 時能獲取最深層的參考資訊。
3.  **架構契合度**：原專案完全遵循 Agent Skills 格式，不需改寫代碼即可整合。

## 4. 實施方案
*   **子模組路徑**：`external/prompt-master`
*   **對接範圍**：映射該目錄作為 Pi 的全域 Skills。
*   **清理**：將原本位於 `pi-skills/prompt-master` 的本地複本移至 `legacy/` 封存。
